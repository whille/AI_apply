# Skill 效果评估方法

> 建立系统化的 skill 效果评估框架，确保技能质量和持续改进

---

## 1. 评估维度

### 1.1 准确性维度 (Accuracy)

| 指标 | 定义 | 测量方法 | 目标值 |
|------|------|----------|--------|
| 结果正确率 | 输出结果符合预期的比例 | 测试用例通过数/总用例数 | ≥ 90% |
| 格式合规率 | 输出格式符合约定的比例 | 格式校验通过数/总输出数 | ≥ 95% |
| 边界处理率 | 边界/异常情况正确处理比例 | 边界用例通过数/边界用例总数 | ≥ 85% |

### 1.2 效率维度 (Efficiency)

| 指标 | 定义 | 测量方法 | 目标值 |
|------|------|----------|--------|
| 执行时间 | skill 完成任务所需时间 | 计时统计（P50/P95） | P95 < 60s |
| Token 消耗 | 单次执行消耗的 token 数 | Claude API 统计 | 按技能类型设定 |
| 交互轮次 | 完成任务需要的对话轮数 | 对话历史计数 | ≤ 5 轮 |
| 重试率 | 需要重试才能成功的比例 | 重试次数/总执行次数 | ≤ 10% |

### 1.3 成本维度 (Cost)

| 指标 | 定义 | 测量方法 | 目标值 |
|------|------|----------|--------|
| 直接成本 | API 调用费用 | Token 数 × 单价 | 记录并监控 |
| 时间成本 | 用户等待时间 | 执行时间统计 | 用户可接受范围 |
| 维护成本 | 更新/修复 skill 的投入 | Git commit/issue 追踪 | 定期回顾 |

### 1.4 用户体验维度 (UX)

| 指标 | 定义 | 测量方法 | 目标值 |
|------|------|----------|--------|
| 首次成功率 | 用户首次使用即成功的比例 | 成功记录/总使用记录 | ≥ 80% |
| 可理解性 | 输出内容易于理解的比例 | 用户反馈/人工评审 | ≥ 85% |
| 可操作性 | 输出可立即执行的比例 | 可执行判定/总输出 | ≥ 90% |

## 2. 测试用例模板

### 2.1 基础模板

```markdown
## 测试用例：[TC-XXX]

### 基本信息
- **所属 Skill**: [skill 名称]
- **用例类型**: 正常流程 / 边界条件 / 异常处理
- **优先级**: P0( blocker ) / P1( critical ) / P2( major ) / P3( minor )
- **场景描述**: [简要描述测试场景]

### 输入数据
```json
{
  "input": "[具体输入数据]",
  "context": "[相关上下文]"
}
```

### 预期输出
```json
{
  "output": "[预期输出]",
  "format": "[格式要求]",
  "constraints": "[约束条件]"
}
```

### 验证点
- [ ] 准确性：结果是否正确
- [ ] 格式：是否符合约定格式
- [ ] 完整性：是否包含所有必要信息
- [ ] 效率：执行时间是否在预期范围

### 执行记录
| 执行时间 | 结果 | 耗时 | Token数 | 备注 |
|----------|------|------|---------|------|
| YYYY-MM-DD | 通过/失败 | Xs | XXX | - |
```

### 2.2 用例类型分类

#### 正常流程用例 (Happy Path)
- 覆盖最常见的使用场景
- 输入数据完整且符合预期
- 验证核心功能正确性

#### 边界条件用例 (Boundary)
- 空输入、极值输入
- 特殊字符、超长输入
- 并发/压力场景

#### 异常处理用例 (Error Handling)
- 无效输入处理
- 资源不可用处理
- 错误恢复能力

#### 回归用例 (Regression)
- 历史缺陷相关场景
- 版本升级验证
- 兼容性验证

## 3. 评估流程

### 3.1 评估周期

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  日常监控   │───→│  定期评估   │───→│  版本评估   │
└─────────────┘    └─────────────┘    └─────────────┘
     实时              每周/月           版本发布时
```

### 3.2 评估步骤

#### Step 1: 准备阶段

```bash
# 1. 确定评估范围
- 明确待评估的 skill 列表
- 确定评估维度和指标
- 准备测试数据集

# 2. 环境就绪
- 确保 Claude Code 环境正常
- 准备必要的工具和依赖
- 清理历史评估数据
```

#### Step 2: 执行评估

```bash
# 1. 运行测试用例
for test_case in test_cases:
    result = execute_skill(test_case.input)
    record_metrics(result)

# 2. 收集指标数据
- 记录执行时间、token 消耗
- 记录成功/失败状态
- 收集用户反馈（如有）

# 3. 异常记录
- 记录失败用例详情
- 记录异常行为
- 记录性能瓶颈
```

#### Step 3: 分析结果

```bash
# 1. 计算指标
accuracy_rate = passed_cases / total_cases
efficiency_score = weighted_average(time_metrics)
cost_per_use = total_cost / use_count

# 2. 对比基准
- 与历史版本对比
- 与目标值对比
- 与同类 skill 对比

# 3. 识别问题
- 分类失败原因
- 定位性能瓶颈
- 分析用户痛点
```

#### Step 4: 输出报告

```markdown
## 评估报告模板

### 概要
- 评估时间：YYYY-MM-DD
- Skill 名称：[名称]
- 评估版本：[版本号]
- 评估结论：通过/需改进/不通过

### 指标汇总
| 维度 | 指标 | 结果 | 目标 | 状态 |
|------|------|------|------|------|
| 准确性 | 结果正确率 | 92% | 90% | ✓ |
| 效率 | P95执行时间 | 45s | 60s | ✓ |
| 成本 | 平均Token数 | 1200 | 1500 | ✓ |

### 问题清单
| 序号 | 问题 | 严重性 | 建议措施 |
|------|------|--------|----------|
| 1 | XXX边界处理失败 | High | 修复边界逻辑 |

### 改进建议
1. [具体改进建议]
2. [具体改进建议]
```

### 3.3 评估等级

| 等级 | 准确率 | 效率 | 综合 | 说明 |
|------|--------|------|------|------|
| **A级** | ≥95% | 优秀 | 可直接使用 | 生产就绪 |
| **B级** | 90-95% | 良好 | 建议使用 | 基本可用，有小问题 |
| **C级** | 85-90% | 一般 | 谨慎使用 | 需要改进后使用 |
| **D级** | <85% | 差 | 禁止使用 | 需要重新设计 |

## 4. 自动化评估工具

### 4.1 评估脚本模板

```bash
#!/bin/bash
# skill-evaluate.sh - Skill 效果评估脚本

SKILL_NAME=$1
TEST_DIR=${2:-"./test_cases/${SKILL_NAME}"}
REPORT_DIR="./reports/${SKILL_NAME}"

echo "=== 评估 Skill: ${SKILL_NAME} ==="
echo "测试用例目录: ${TEST_DIR}"
echo "报告输出目录: ${REPORT_DIR}"

# 统计变量
TOTAL=0
PASSED=0
FAILED=0
TOTAL_TIME=0
TOTAL_TOKENS=0

# 执行测试
for test_file in ${TEST_DIR}/*.json; do
    if [ -f "$test_file" ]; then
        TOTAL=$((TOTAL + 1))
        echo "执行: $test_file"

        # 记录开始时间
        START_TIME=$(date +%s)

        # 执行 skill (实际调用方式根据 skill 类型调整)
        # RESULT=$(claude-code --skill ${SKILL_NAME} --input "${test_file}")

        # 记录结束时间
        END_TIME=$(date +%s)
        ELAPSED=$((END_TIME - START_TIME))
        TOTAL_TIME=$((TOTAL_TIME + ELAPSED))

        # 判断结果 (简化示例)
        if [ $? -eq 0 ]; then
            PASSED=$((PASSED + 1))
            echo "  结果: 通过 (${ELAPSED}s)"
        else
            FAILED=$((FAILED + 1))
            echo "  结果: 失败"
        fi
    fi
done

# 输出统计
echo ""
echo "=== 评估结果 ==="
echo "总用例数: ${TOTAL}"
echo "通过数: ${PASSED}"
echo "失败数: ${FAILED}"
ACCURACY=$(echo "scale=2; ${PASSED}*100/${TOTAL}" | bc)
echo "准确率: ${ACCURACY}%"
echo "平均时间: $((TOTAL_TIME / TOTAL))s"
```

## 5. 持续改进机制

### 5.1 数据驱动改进

```
收集数据 → 分析问题 → 制定改进 → 验证效果 → 部署上线
    ↑                                              ↓
    └──────────────── 反馈循环 ←───────────────────┘
```

### 5.2 改进优先级

| 优先级 | 触发条件 | 响应时间 |
|--------|----------|----------|
| P0 | 准确率 < 80% 或核心功能失效 | 24小时内 |
| P1 | 准确率 < 90% 或性能下降 > 50% | 3天内 |
| P2 | 用户体验问题或效率不达标 | 1周内 |
| P3 | 优化建议或小改进 | 迭代计划中 |

### 5.3 版本管理

```bash
# Skill 版本号规则
MAJOR.MINOR.PATCH

- MAJOR: 重构或不兼容改动
- MINOR: 新增功能或显著改进
- PATCH: Bug 修复或小优化

# 评估结果与版本关联
SKILL.md 中记录:
- 当前版本评估结果
- 历史版本评估记录
- 已知问题和限制
```

### 6.2 常见问题处理

| 问题 | 可能原因 | 处理方法 |
|------|----------|----------|
| 准确率波动 | 测试数据不稳定 | 增加样本量，使用确定性测试 |
| Token 超标 | Prompt 过长或输出过多 | 优化 Prompt，限制输出长度 |
| 响应超时 | 复杂任务或网络问题 | 拆分任务，增加超时设置 |
| 格式错误 | Prompt 约束不足 | 强化格式约束，添加示例 |
