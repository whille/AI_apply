# AI Agent 使用分享

### 怎么算用好？

- 长任务：能否持续执行不丢失上下文？
- 复杂大项目：能否理解整体架构？
- 并发：能否多任务并行(worktree)？
- 可验证：有没test cases 对比验证效果?

### 项目复杂度与效率衰减

随着项目复杂度增加（文件数、代码行数、耦合度），Agent 效率非线性衰减，复杂项目需要人深度参与。

### Agent 上限表现

**1. 上下文窗口限制**
- 简单项目：代码 + 文档 < 50K tokens → Agent 轻松处理
- 复杂项目：代码 + 文档 > 200K tokens → 需要分段处理、信息丢失

**2. 全局理解困难**
```
修改模块 A → 影响模块 B（Agent 不知道）→ 引入 bug → 发现修复 → 又影响模块 C → 循环...
```

**3. 耦合度高时的连锁反应**
- 单次改动 → 触发多处修改 → 返工成本指数增长

## 三、怎么用 Agent？

### 学习用 Agent

做稍微复杂的实践练习：不是俄罗斯方块这种LLM一次能完成的，比如微信小程序，[麻将网页版](https://whille.github.io/mahjong)（熟悉、但规则不容易一次整理完），做 vibe coding，总结经验。

### Vibe Coding

**核心理念**：构建即理解（Build to understand）

```
传统开发：设计 → 编码 → 测试
Vibe Coding：原型 → 运行 → 感受 → 调整 → 再原型
```

**人机协作分工**：

| 角色 | 负责内容 |
|------|----------|
| **人** | 领域知识（规则、策略）、目标设定 |
| **Agent** | 边界发现、代码实现、异常处理 |

### 注意事项

#### 1. 业务理解必须提前计划好

**错误流程**：
给 Agent 简单描述 → Agent 实现 → 发现不对 → 反复修改 → 效率骤降

**正确流程**：
业务说明（写清楚）→ 举例说明（输入输出）→ Agent 实现 → 验证 → 微调

**为何重要**：
- 规划投入 1 小时，可能节省 5 小时返工
- 业务理解不清晰，Agent 效率优势被抵消

#### 2. 术语对齐避免偏差

老项目的中英文术语，人机理解可能有偏差，建议项目开始前先对齐。

#### 3. 规则矛盾需人发现

**实际案例**：Gurobi 优化 转码水位规则

- 产品给的规则隐含着矛盾不易发现
- 最终修改设计规则
- 经验教训：业务理解要提前计划好：业务说明、举例要写清楚，否则反复修改效率明显降低

#### 4. 未来借鉴 openclaw 工作方式 到业务：
 每日定时任务:

- 定期代码review
- 提交重构 PR
- 分析运行日志

## 五、理解 Claude Code原理

### 途径一：简化版实现

> 示例：https://github.com/shareAI-lab/learn-claude-code

**优势**：代码量小，易于理解核心机制，可本地运行调试

**学习要点**：
1. 理解核心循环：感知 → 决策 → 执行
2. 理解工具调用机制
3. 理解上下文管理
4. 回到完整版，对比差异

### 途径二：源码深度学习

### 我的理解

**关于 context**：
- MCP 不适合传输大量data → 用 skill（grep/py 等）多维度灵活分析
- 常用 LLM prompt 组合 → 创建 skill（增加确定性、减少关键词、减少 context）
- 信息层次化：`@target file` 代替具体内容，skill 代替模糊的 LLM 交互

### 趋势判断

| 现在 | 未来 | 原因 |
|------|------|------|
| 拖拽式工作流（Coze） | skill 化 | 复杂流程人工操作多，工程化更利于改进 |
| RAG 工程 | hermes 进化 | 从静态检索到自改进学习循环 |
