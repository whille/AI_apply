# AI Agent 应用

## AI 应用的上限(怎么算用好)？

- 长任务：能否持续执行不丢失上下文？
- 复杂大项目：能否理解整体架构？
- 并发：能否多任务并行？
- 高质量：输出是否可靠、可复现？
- 固化 skill → AI 进化：能否从经验中学习？

---

## 思路从哪来？

- 创造性思维 + meta 方法论
- 避免机械性记忆，注重理解与应用
- LLM →（固化）skill/wiki →（重构）工程化

### 趋势会往哪走？

**个人判断**：

| 现在 | 未来 | 原因 |
|------|------|------|
| 拖拽式工作流（Coze） | skill 化 | 复杂流程人工操作多，工程化更利于改进 |
| RAG 工程 | hermes 进化 | 从静态检索到自改进学习循环 |

#### Coze 拖拽 vs Claude Skill 对比

| 维度 | Coze 拖拽工作流 | Claude Skill |
|------|----------------|--------------|
| **上手门槛** | 低，可视化拖拽 | 中，需懂代码/prompt |
| **调试** | 即时预览，直观 | 需运行验证，迭代调试 |
| **版本控制** | 平台内，不透明 | Git 管理，可追溯 |
| **复用性** | 依赖平台导出 | 纯文本，跨项目复用 |
| **协作** | 平台内分享 | Git 协作，代码 review |
| **灵活性** | 受限于组件 | 自由扩展，可调用任意工具 |
| **维护成本** | 低频修改：方便 | 高频迭代：工程化优势 |
| **迁移风险** | 绑定平台 | 开放格式，无锁定 |

**结论**：

- 简单场景、非技术人员 → Coze 够用
- 复杂流程、持续迭代 → Skill 工程化
- 趋势：高频使用的流程会从拖拽迁移到代码化

## 手段有哪些？

### Skill：丰富的工具库

- 我的需求有没有优秀的实现？没有就自制
- 自制原则：复用优先

### Rules & Hooks：精简的场景配置

- 规则精简，hooks 轻量
- 避免过度工程化

### MCP：必要的能力

**必要**：搜索能力（websearch）、图文理解

**按需**：playwright 等

### 研发 Agent 怎么选？

**Cursor vs Claude (Kscc)**：

| 维度 | Cursor | Claude (Kscc) |
|------|--------|---------------|
| IDE 功能 | 丰富 | 依赖编辑器配合 |
| Context 管理 | 方便，信息量大 | 需主动管理，更聚焦 |
| 自动化 | 受限 | 灵活 |
| 并行开发 | 不支持 worktree | 支持 worktree 隔离 |

**设计哲学差异**：

| 产品 | 定位 | 设计取舍 |
|------|------|----------|
| Cursor | 聚焦 IDE 文件编写 | 不支持 worktree，避免用户混乱 |
| Claude | 通用 Agent | 开放能力，用户自行组织 |

**Claude 补短板方案**：

```
编辑器交互（vim/VSCode）
    +
tmux/iTerm 分屏
    → 弥补 IDE 功能短板
    → 关注重点，避免信息过载
```

## 典型应用怎么跑？

### 如何搜索知识？

```bash
/intel --deep "关键词" --sources bilibili,github,rss
```

**intel 技能**：信息源跟踪系统 [源码](https://github.com/whille/myclaude)
- 多源并发扫描、去重、评分
- 输出结构化报告：总结 + 洞察

### 如何快速学习？

人机结合学习法：

```
循环开始
  人机分析效果 → agent 对话调整 → 改进规则
       ↑                              │
       └──────────── 未达成目标 ───────┘
                    达成 → 结束
```

**示例**：精简 Gitbook，通过问答测试判断知识储备，有限次循环后获得 80%+ 有用信息。

### 羲和在线效果分析

**两种手段对比**：

| 手段 | 数据量 | 查询方式 | 效果 |
|------|--------|----------|------|
| 结构化日志 | 维度较多 | DB 查询 | 有限，依赖预设维度 |
| log 日志 | 100M+/天 | agent + bash | 更灵活，智能分析 |

**结论**：

```
先用 agent 解决问题
      ↓
沉淀经验
      ↓
固化成 skill + 日志补充
```

## 如何评估效果？

同一任务，相同 LLM，对比两种实现：

```
方案 A vs 方案 B
维度：时间、准确率、代码量、可复现性
```

### 示例：知识库格式选型

对比多种知识管理方案：

| 方案 | 复杂度 | 与 LLM 交互 |
|------|--------|-------------|
| Markdown + grep | 低 | 直接 |
| llm-wiki | 中 | 需消化流程 |
| Graphify（知识图谱） | 高 | 需转换 |
| Obsidian | 中 | 需导出 |

**结论**：
- Claude 用 grep 搜索 md 文件，效果与复杂方案相当
- md 格式足够，无需复杂化

## 业务应用设想

### 从哪里开始？

从严格程度逐步推进：

| 阶段 | 应用 | 风险 |
|------|------|------|
| 低 | 辅助分析 | 影响 limited |
| ↓ | bug 排查 | 有验证手段 |
| ↓ | 代码 PR | 人工 review |
| 高 | 功能开发 | 需要测试覆盖 |

### 团队集智如何做？

```
技能固化共享 → 团队复用最佳实践
      ↓
记忆跟踪 → 不丢失历史上下文
      ↓
智能分析反馈 → 识别问题模式
      ↓
技能优化 → 迭代改进
      ↓
重构决策 → 数据驱动
```
