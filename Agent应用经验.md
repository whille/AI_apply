# AI Agent 应用

## 核心理念：人机结合

### 分工原则

```
┌─────────────────────────────────────────────────────────┐
│                    人机协作循环                          │
├─────────────────────────────────────────────────────────┤
│                                                         │
│   Agent 能力                 人能力                     │
│   ─────────                  ─────                      │
│   收集信息        ──────>     决策判断                    │
│   整理知识        ──────>     洞察提炼                    │
│   辅助学习        <──────     经验传授                    │
│   执行落地        <──────     目标设定                    │
│                                                         │
└─────────────────────────────────────────────────────────┘
                          ↓
                    技能进化循环
```

### Agent 擅长的

| 能力 | 具体表现 | 价值 |
|------|----------|------|
| **信息收集** | 多源扫描、去重、筛选 | 节省时间、覆盖全面 |
| **知识整理** | 结构化、关联、索引 | 便于检索、理解 |
| **辅助学习** | 提问测试、概念解释 | 加速掌握、发现盲点 |
| **执行落地** | 代码生成、测试运行 | 快速验证、迭代 |

### 人擅长的

| 能力 | 具体表现 | 价值 |
|------|----------|------|
| **决策判断** | 方向选择、风险权衡 | 把握大局、承担责任 |
| **洞察提炼** | 模式识别、本质把握 | 超越数据、形成智慧 |
| **经验传授** | 领域知识、隐性规则 | 补充 Agent 盲区 |
| **目标设定** | 需求定义、价值判断 | 引导方向、衡量结果 |

### 协作循环

```
循环一：学习循环
┌──────────────────────────────────────────┐
│                                          │
│   Agent 整理知识 → 人学习理解              │
│        ↑                    ↓            │
│   人补充盲点 ← 人发现问题反馈              │
│                                          │
└──────────────────────────────────────────┘

循环二：决策循环
┌──────────────────────────────────────────┐
│                                          │
│   Agent 提供选项 → 人判断决策              │
│        ↑                    ↓            │
│   Agent 执行 ← 人设定目标                 │
│                                          │
└──────────────────────────────────────────┘

循环三：进化循环
┌──────────────────────────────────────────┐
│                                          │
│   Agent 辅助学习 → 人能力提升              │
│        ↑                    ↓            │
│   Skill 优化 ← 人沉淀经验                 │
│                                          │
└──────────────────────────────────────────┘
```

### 核心公式

$$
\text{人机结合} = \underbrace{\text{Agent 执行}}_{\text{效率}} + \underbrace{\text{人决策}}_{\text{方向}} + \underbrace{\text{技能进化}}_{\text{积累}}
$$

### 实践原则

| 原则 | 错误做法 | 正确做法 |
|------|----------|----------|
| **人主导决策** | 让 Agent 替你判断 | Agent 提供选项，人决策 |
| **Agent 辅助执行** | 事必躬亲 | 让 Agent 处理重复、机械工作 |
| **经验持续沉淀** | 每次从零开始 | 固化成 Skill，复用经验 |
| **能力边界清晰** | 期望 Agent 做不到的事 | 理解 Agent 能力，合理分工 |

### 技能进化路径

```
实践 → 发现问题 → 解决问题 → 沉淀经验 → 固化 Skill → 复用改进
  ↑                                                        │
  └────────────────────── 新实践 ──────────────────────────┘
```

**固化方式**：

| 经验类型 | 固化形式 | 示例 |
|----------|----------|------|
| 检索流程 | Skill | `/intel --deep` |
| 知识库 | Wiki/Markdown | `llm-wiki` |
| 开发流程 | Skill + Hooks | `/commit`, `/review-pr` |
| 学习方法 | Skill | `/book-ingest`, `/knowledge-test` |

### 终极目标

$$
\text{目标} = \underbrace{\text{人越来越强}}_{\text{学习}} + \underbrace{\text{Agent 越来越聪明}}_{\text{进化}} + \underbrace{\text{协作越来越顺}}_{\text{磨合}}
$$

**三个越来越**：
1. **人越来越强**：Agent 辅助学习，加速成长
2. **Agent 越来越聪明**：经验固化，能力累积
3. **协作越来越顺**：默契形成，效率提升

## AI 应用的上限(怎么算用好)？

- 长任务：能否持续执行不丢失上下文？
- 复杂大项目：能否理解整体架构？
- 并发：能否多任务并行？
- 高质量：输出是否可靠、可复现？
- 固化 skill → AI 进化：能否从经验中学习？

### 趋势会往哪走？

**个人判断**：

| 现在 | 未来 | 原因 |
|------|------|------|
| 拖拽式工作流（Coze） | skill 化 | 复杂流程人工操作多，工程化更利于改进 |
| RAG 工程 | hermes 进化 | 从静态检索到自改进学习循环 |

#### Coze 拖拽 vs Claude Skill 对比

| 维度 | Coze 拖拽工作流 | Claude Skill |
|------|----------------|--------------|
| **上手门槛** | 低，可视化拖拽 | 中，需懂代码/prompt |
| **调试** | 即时预览，直观 | 需运行验证，迭代调试 |
| **版本控制** | 平台内，不透明 | Git 管理，可追溯 |
| **复用性** | 依赖平台导出 | 纯文本，跨项目复用 |
| **协作** | 平台内分享 | Git 协作，代码 review |
| **灵活性** | 受限于组件 | 自由扩展，可调用任意工具 |
| **维护成本** | 低频修改：方便 | 高频迭代：工程化优势 |
| **迁移风险** | 绑定平台 | 开放格式，无锁定 |

**结论**：

- 简单场景、非技术人员 → Coze 够用
- 复杂流程、持续迭代 → Skill 工程化
- 趋势：高频使用的流程会从拖拽迁移到代码化


### Gurobi 优化模型开发

**项目特点**：
- 数学优化模型（约束、目标函数）
- 业务规则复杂、耦合度高
- 中小规模模型

**Agent 表现**：

| 维度 | 评价 | 说明 |
|------|------|------|
| 代码实现 | 高效 | 快速实现约束、目标函数 |
| 调试排错 | 良好 | 能定位不可行解、无界解 |
| 性能优化 | 一般 | 需人指导优化方向 |

**关键经验**：

#### 1. 业务理解必须提前计划好

```
错误流程：
给 Agent 简单描述 → Agent 实现 → 发现不对 → 反复修改 → 效率骤降

正确流程：
业务说明（写清楚） → 举例说明（输入输出） → Agent 实现 → 验证 → 微调
```

**业务说明模板**：

```markdown
## 业务背景
[一句话说明]

## 核心规则
1. 规则一：具体描述
2. 规则二：具体描述
   - 子规则 2.1
   - 子规则 2.2

## 示例
输入：xxx
期望输出：xxx
实际业务含义：xxx

## 边界情况
- 情况一：处理方式
- 情况二：处理方式
```

**为何重要**：
- Agent 对业务理解依赖输入质量
- 描述模糊 → 理解偏差 → 返工成本高
- 提前投入时间规划，整体效率更高

#### 2. 规则矛盾难以发现

**实际案例**：转码水位规则

```
规则 A：转码后水位不能超过阈值
规则 B：某些场景必须转码（策略要求）
约束 C：存储容量有限

→ 实际运行发现：三个规则在某些情况下无法同时满足
→ 最终：修改业务规则，放宽约束
```

**Agent 的局限**：
- Agent 会尝试满足所有约束
- 但不会主动质疑规则是否合理
- 需要人从业务角度发现矛盾

**建议流程**：

```
建模前：
  业务规则审查 → 发现潜在冲突 → 提前解决

建模后：
  小数据测试 → 检查不可行情况 → 分析是否规则问题

上线前：
  真实数据验证 → 发现边界问题 → 确认是否需要修改规则
```

#### 3. 开发效率曲线

```
效率
  │
  │     ┌───────┐
  │     │ 高效率│
  │ ┌───┘       └───┐
  │ │               │
  │ │               └───┐
  │ │                   │ 低效率
  │ └───────────────────┘ (反复修改)
  │
  └─────────────────────────→ 时间
    规划  实现  调试  返工
```

**结论**：
- 规划投入 1 小时，可能节省 5 小时返工
- 业务理解不清晰，Agent 效率优势被抵消
- 复杂模型尤其需要前期规划

### 老项目术语对齐

**问题**：中英文术语理解偏差

| 场景 | 问题 | 后果 |
|------|------|------|
| 老项目代码 | 中文注释 + 英文变量名 | Agent 理解不一致 |
| 业务文档 | 专业术语缩写 | Agent 无法准确理解 |
| 数据库字段 | 字段名含义模糊 | 生成的代码语义错误 |

**示例**：

```
业务术语："水位"
  - 水文学：水位高度
  - 存储：存储容量使用率
  - 转码：转码队列积压量

Agent 如果不理解上下文 → 错误实现
```

**解决方案**：

#### 术语对齐文档

```markdown
# 项目术语表

| 术语 | 英文 | 含义 | 示例 |
|------|------|------|------|
| 水位 | water_level | 存储容量使用率 | "水位达到80%" |
| 转码 | transcoding | 视频格式转换 | "转码任务" |
| 命中率 | hit_rate | 缓存命中率 | "命中率95%" |
```

#### 使用方式

```
项目开始前：
  创建术语表 → 提供给 Agent

开发过程中：
  新术语及时补充 → 保持术语表更新

团队协作：
  术语表放在项目根目录 → 统一理解
```

**收益**：
- 减少理解偏差导致的返工
- 提高代码语义一致性
- 便于团队知识传承

### 复杂项目的 Agent 上限

#### 问题：非线性效率衰减

```
效率
  │
  │\
  │ \
  │  \
  │   \
  │    \  ← 线性衰减（简单项目）
  │     \
  │      \___
  │          \___  ← 非线性衰减（复杂项目）
  │              \___
  └─────────────────────→ 复杂度
    简单  中等  复杂  非常复杂
```

**原因分析**：

| 因素 | 简单项目 | 复杂项目 |
|------|----------|----------|
| 代码耦合度 | 低 | 高（改一处影响多处） |
| 业务逻辑复杂度 | 单一路径 | 多路径、多分支 |
| 上下文理解 | 局部即可 | 需要全局理解 |
| 测试覆盖 | 容易 | 困难（场景组合爆炸） |

#### Agent 上限表现

**1. 上下文窗口限制**

```
简单项目：代码 + 文档 < 50K tokens → Agent 轻松处理
复杂项目：代码 + 文档 > 200K tokens → 需要分段处理、信息丢失
```

**2. 全局理解困难**

```
修改模块 A
  ↓
影响模块 B（Agent 不知道）
  ↓
引入 bug
  ↓
发现、修复
  ↓
又影响模块 C
  ↓
循环...
```

**3. 耦合度高时的连锁反应**

```
单次改动 → 触发多处修改 → 返工成本指数增长
```

#### 实测数据（未使用业内领先 LLM）

| 项目复杂度 | 文件数 | 代码行数 | Agent 效率 |
|------------|--------|----------|-----------|
| 简单 | 1-5 | < 1000 | ⭐⭐⭐⭐⭐ |
| 中等 | 5-20 | 1000-5000 | ⭐⭐⭐⭐ |
| 复杂 | 20-50 | 5000-20000 | ⭐⭐⭐ |
| 非常复杂 | > 50 | > 20000 | ⭐⭐（需要人深度参与） |

**注**：使用 Claude Opus/Sonnet 测试，未使用 GPT-4o 等其他模型对比。

#### 应对策略

**策略一：降低复杂度**

```
大项目 → 拆分成小模块 → 分模块开发 → 集成
```

**策略二：加强规划**

```
复杂项目必须：
  - 详细设计文档
  - 接口定义清晰
  - 测试用例先行
  - 分阶段实现
```

**策略三：人机分工调整**

| 项目阶段 | Agent 负责 | 人负责 |
|----------|-----------|--------|
| 需求分析 | 辅助整理 | 主导理解 |
| 架构设计 | 提供选项 | 主导决策 |
| 编码实现 | 主导执行 | Review |
| 联调集成 | 辅助调试 | 主导排查 |
| 上线验证 | 执行测试 | 分析结果 |

**策略四：适时引入更强工具**

```
Agent 能力上限 →
  考虑：
  - 更强的 LLM（Opus、GPT-4o）
  - 专用工具（IDE、调试器）
  - 专家介入
```

## 复杂项目如何上手？

### 项目复杂度判断

| 类型 | 特征 | 示例 | Agent 策略 |
|------|------|------|-----------|
| 简单 | 单文件、逻辑清晰 | 俄罗斯方块、计算器 | 一次性完成 |
| 中等 | 多文件、需迭代 | 工具类小程序 | 分模块开发 |
| 复杂 | 规则多、交互复杂 | 麻将、棋牌游戏 | Vibe Coding |

### 什么是 Vibe Coding?

> 来源：Andrej Karpathy, "Building Software in the Era of LLMs"

**核心理念**：构建即理解（Build to understand）

```
传统开发：设计 → 编码 → 测试
Vibe Coding：原型 → 运行 → 感受 → 调整 → 再原型
```

**与 TDD 对比**：

| 维度 | TDD | Vibe Coding |
|------|-----|-------------|
| 起点 | 测试用例 | 最小可运行原型 |
| 反馈 | 测试通过/失败 | 运行时感受 |
| 迭代 | 红-绿-重构 | 跑-感-改-跑 |
| 适用 | 逻辑明确 | 规则复杂、边界模糊 |

### 实践流程：麻将网页版开发

**项目特点**：
- 规则熟悉但不易完整梳理
- 交互细节需要运行时验证
- 边界情况多（胡牌判定、番型计算）

#### Step 1: 最小可运行原型（MVP）

```bash
# 目标：能跑起来，不管规则是否完整
Agent 任务："实现麻将基础流程：发牌、摸牌、打牌、四人轮流"
```

**产出**：
- 基础游戏循环
- 简单牌桌 UI
- 基本交互（点击出牌）

#### Step 2: 运行感受，发现缺漏

```
实际运行 → 发现问题：
- "碰"的逻辑没有
- 胡牌判定不完整
- 番型计算太简单
```

#### Step 3: 增量补充规则

```bash
# 每次添加一个功能模块
Agent 任务："添加碰、杠、吃逻辑"
→ 测试运行

Agent 任务："完善胡牌判定，支持七对、清一色"
→ 测试运行
```

#### Step 4: 边界发现循环

```
发现边界 → 记录规则 → 补充代码 → 再验证
     ↑                                      │
     └────────── 新问题 ──────────────────┘
```

**示例**：
- 运行发现："杠了之后摸牌算不算自摸？"
- Agent 分析边界条件，补充规则
- 记录到 `mahjong_rules.md`
- 下次直接查阅，无需再问

#### Step 5: 人机协作分工

| 角色 | 负责内容 |
|------|----------|
| **人** | 领域知识（规则、策略）、目标设定 |
| **Agent** | 边界发现、代码实现、异常处理 |

**最终产出**：
- 可玩的麻将网页版
- 完整规则文档（`mahjong_rules.md`）
- Skill 模板（可用于其他棋牌游戏）
- 测试用例集

### 实践流程：微信小程序开发

#### 特点

- 需求变化快
- UI 交互需要实际体验
- 平台限制多（审核、API）

#### Vibe Coding 应用

```
快速原型 → 真机体验 → 发现体验问题 → 调整 → 再体验
```

**示例：电商小程序**

```bash
# 迭代 1：购物车流程
Agent："实现：浏览商品 → 加购 → 查看购物车"
→ 真机体验，发现购物车交互不顺

# 迭代 2：优化交互
Agent："购物车滑动删除，左滑显示删除按钮"
→ 体验流畅

# 迭代 3：支付集成
Agent："集成微信支付"
→ 测试流程，补充异常处理
```

#### 关键经验

| 阶段 | 要点 |
|------|------|
| 原型验证 | 优先跑通核心流程 |
| 功能增加 | 每次一个模块，验证后再加 |
| UI 调整 | 需真机体验，截图反馈 Agent |
| 性能优化 | 最后做，先跑起来再优化 |

### 经验总结

#### 1. 迭代节奏控制

```
快速原型（1-2 小时）
    ↓
感受体验（发现问题）
    ↓
精准补充（不大改，只修问题）
    ↓
再体验 → 循环
```

**错误做法**：
- ❌ 一次让 Agent 写完整系统
- ❌ 不运行就继续加功能
- ❌ 完美主义，追求一次到位

**正确做法**：
- ✅ MVP 优先，先跑起来
- ✅ 边跑边补，感受驱动
- ✅ 记录规则，用例补充

#### 2. 规则文档沉淀

**动态维护规则文档**：

```markdown
# mahjong_rules.md

## 基础规则
- 四人轮流摸牌打牌
- 可碰、可杠、可吃

## 边界情况（运行中发现）
- 杠后摸牌：算自摸
- 最后四张：不能碰杠
- 流局判定：剩牌 ≤ 14 张
```

**价值**：
- Agent 下次直接查阅
- 减少重复解释成本
- 可复用到其他项目

#### 3. Skill 提炼

复杂项目完成后，提炼通用 skill：

```bash
/skill-create
# 示例：从麻将项目提炼
→ 棋牌游戏开发模板
→ 规则文档结构
→ 边界发现流程
```

#### 4. 失败模式识别

| 失败模式 | 现象 | 解决方案 |
|----------|------|----------|
| 规则不一致 | 不同版本规则冲突 | 统一规则文档 |
| 上下文丢失 | 迭代多次后忘记前面约定 | 用文档/注释记录 |
| 过度设计 | 添加未要求功能 | 明确 MVP 范围 |
| 体验缺失 | 只跑通逻辑，不考虑交互 | 真机测试，感受反馈 |

#### 5. 项目知识管理

```
项目文档（项目内）
    ↓ 提炼
通用 Skill（全局复用）
    ↓ 积累
经验库（方法论）
```

**关键点**：
- 边缘案例优先记录
- 异常流程必须文档化
- 每个项目都应有"踩坑记录"

### 工程化建议

#### 不要过度工程化

- Vibe Coding ≠ 无设计
- 最小设计 + 快速迭代
- 设计要服务于迭代

#### 保持灵活性

```
预判变化的模块 → 抽象
稳定的部分 → 直接写
```

#### 关注用户价值

- 先跑通核心流程
- 再优化边缘场景
- 不要追求完美

## 如何评估效果？

同一任务，相同 LLM，对比两种实现：

```
方案 A vs 方案 B
维度：时间、准确率、代码量、可复现性
```

### 示例：知识库格式选型

对比多种知识管理方案：

| 方案 | 复杂度 | 与 LLM 交互 |
|------|--------|-------------|
| Markdown + grep | 低 | 直接 |
| llm-wiki | 中 | 需消化流程 |
| Graphify（知识图谱） | 高 | 需转换 |
| Obsidian | 中 | 需导出 |

**结论**：
- Claude 用 grep 搜索 md 文件，效果与复杂方案相当
- md 格式足够，无需复杂化

## 业务应用设想

### 从哪里开始？

从严格程度逐步推进：

| 阶段 | 应用 | 风险 |
|------|------|------|
| 低 | 辅助分析 | 影响 limited |
| ↓ | bug 排查 | 有验证手段 |
| ↓ | 代码 PR | 人工 review |
| 高 | 功能开发 | 需要测试覆盖 |

### 团队集智如何做？

```
技能固化共享 → 团队复用最佳实践
      ↓
记忆跟踪 → 不丢失历史上下文
      ↓
智能分析反馈 → 识别问题模式
      ↓
技能优化 → 迭代改进
      ↓
重构决策 → 数据驱动
```
