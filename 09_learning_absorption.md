# 高效学习与吸收

> 快速掌握 GitHub 项目、技术书籍，结构化沉淀可复用知识

## 问题清单

- 如何快速理解一个 GitHub 项目？
- 如何提取 insight 并验证价值？
- 学习成果如何结构化沉淀？
- 人机共用的资料怎么组织？

---

## 核心流程

```
输入 → 快速理解 → 提取 Insight → 验证价值 → 结构化沉淀
                    ↓                ↓           ↓
                 判断是否值得     测试验证     md 落盘
                    ↓                ↓           ↓
                 决策：深入/放弃   确认能力边界  AI 可复用
```

## 场景一：GitHub 项目学习

### 目标

1. **快速总结**：30 分钟内理解项目核心
2. **Insight 提取**：发现独特价值点
3. **优点判断**：是否值得深入？
4. **测试验证**：设计 case 验证核心能力

### 执行步骤

#### Step 1: 项目扫描（5-10 min）

```
输入：GitHub URL
输出：项目结构概览

动作：
1. 分析 README → 核心定位
2. 扫描目录结构 → 模块划分
3. 查看 package.json / requirements.txt → 技术栈
4. 检查 Star/Issue/更新频率 → 活跃度
```

#### Step 2: 核心代码阅读（10-15 min）

```
输入：关键文件列表
输出：核心逻辑理解

动作：
1. 入口文件 → 启动流程
2. 核心模块 → 关键算法/设计模式
3. 测试文件 → 能力边界
```

#### Step 3: Insight 提取（5-10 min）

```
输出：Insight 清单

模板：
### Insight: [核心发现]
- **价值点**：解决了什么问题？
- **创新点**：与同类项目对比
- **适用场景**：什么情况下用？
- **风险点**：限制和不足
```

#### Step 4: 价值判断

```
决策矩阵：
| 维度 | 权重 | 评分(1-5) |
|------|------|-----------|
| 解决痛点清晰度 | 30% | |
| 实现简洁度 | 20% | |
| 可扩展性 | 20% | |
| 社区活跃度 | 15% | |
| 文档质量 | 15% | |

总分 ≥ 4：值得深入
总分 2-4：选择性借鉴
总分 < 2：放弃
```

#### Step 5: 测试验证设计

```
目标：验证核心优点的有效性

测试用例设计原则：
1. 最小化：1-3 个 case 即可
2. 针对性：针对宣称的核心优点
3. 可对比：与现有方案对比

模板：
### 测试 Case: [验证什么]
- **假设**：项目宣称的能力
- **测试方法**：具体步骤
- **预期结果**：应该看到什么
- **对比基准**：替代方案的结果
```

### 输出模板

```markdown
# 项目学习笔记：[项目名]

## 基本信息
- URL：
- Stars：
- 更新频率：
- 技术栈：

## 核心定位
[一句话概括]

## 架构理解
[核心模块图]

## Insight 清单
### Insight 1: [标题]
- 价值点：
- 创新点：
- 适用场景：
- 风险点：

## 价值判断
| 维度 | 评分 | 说明 |
|------|------|------|
| 痛点清晰度 | | |
| 实现简洁度 | | |
| ... | | |
| **总分** | | |

## 验证计划
### Case 1: [标题]
- 假设：
- 方法：
- 预期：
- 基准：

## 后续行动
- [ ] 深入 / 选择性借鉴 / 放弃
- [ ] 如果深入：学习路径
```

## 场景二：技术书籍/知识学习

### 目标

1. **高效掌握**：提取核心概念和框架
2. **Insight 沉淀**：个人理解和延伸
3. **可复用**：AI 可基于笔记回答问题

### 执行步骤

#### Step 1: 结构预览

```
动作：
1. 目录 → 知识框架
2. 前言 → 作者意图
3. 核心章节 → 重点标记
```

#### Step 2: 分层阅读

```
层级：
L1: 核心概念（必须掌握）
L2: 方法论（需要理解）
L3: 案例/细节（可查可略）

策略：
- L1 精读 + 笔记
- L2 理解 + 总结
- L3 快速浏览，需要时回头
```

#### Step 3: Insight 提取

```
模板：
### 核心概念：[概念名]
- 定义：
- 为什么重要：
- 与其他概念关系：
- 个人理解：
- 应用场景：
```

## 人机共用设计

### 资料组织原则

```
AI_apply/
├── project_notes/          # 项目学习笔记
│   ├── [project_name].md
│   └── ...
├── book_notes/             # 书籍学习笔记
│   ├── [book_name].md
│   └── ...
└── insights/                # 跨项目洞察
    ├── [topic].md
    └── ...
```

### 笔记规范

```yaml
# 每个笔记必须包含：
1. 元信息块（来源、时间、状态）
2. 核心内容（结构化）
3. Insight 清单（个人见解）
4. 待办/行动项

# 便于 AI 理解：
- 使用标准标题层级
- 关键概念加粗
- 列表结构化
- 有明确的"个人理解"段落
```

## 工具调研结果

### GitHub 项目分析工具

| 工具 | 类型 | 功能 | 可用性 |
|------|------|------|--------|
| **Repomix** | CLI | 仓库打包为 txt/xml/md，含 Token 计数 | ✅ 立即可用 `npx repomix` |
| **GitMCP** | MCP | `gitmcp.io/{owner}/{repo}` 零配置访问文档 | ✅ 立即可用 |
| **SourceSage** | Skill | 源码转 AI 友好 Markdown | ⚠️ 需安装 |
| **RepoLens** | Engine | AST 解析、依赖图、模块提取 | ⚠️ 需集成 |

### 推荐组合

```
GitHub 项目学习流程：
1. GitMCP 获取文档概览（零配置）
2. Repomix 打包核心代码（token 可控）
3. Claude 分析 + 提取 Insight
4. 输出到 project_notes/
```

### 知识库构建工具

| 工具 | URL | 特点 |
|------|-----|------|
| **llm-wiki-agent** | github.com/SamurAIGPT/llm-wiki-agent | 自动构建互链 Wiki |
| **karpathy-llm-wiki** | github.com/Astro-Han/karpathy-llm-wiki | Agent Skills 兼容 |
| **llm-wiki (中文)** | github.com/luotwo/llm-wiki | 中文文档，含 Skill |

**llm-wiki 核心理念**：
- LLM 作为知识编译器：raw 文档进，结构化 Wiki 出
- 不用向量数据库，纯 Markdown + Obsidian
- 知识持续积累，复利效应

### Gap 分析

| 能力 | 现有工具 | Gap |
|------|----------|-----|
| 项目结构分析 | GitMCP + Repomix | ✅ 足够 |
| Insight 提取 | Claude 原生 | ✅ 足够 |
| 测试 Case 设计 | Claude 原生 | ⚠️ 需 Skill 引导 |
| 知识沉淀 | llm-wiki | ✅ 足够 |
| 视频内容学习 | — | ⏸️ 暂缓（人消耗大） |
